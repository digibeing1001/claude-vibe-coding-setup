# 三层嵌套反馈架构

整套 Loop Engineering 是三个不同时间尺度的嵌套控制系统，不是并列流程。控制逻辑用航海隐喻最清晰：

- **L1（最内层）** = 自动驾驶和船员：每隔几分钟按当前航海图（Spec+Evals）调帆绳、查罗盘，让船在当前风向下稳定不触礁
- **L2（中间层）** = 船长：每隔几小时登驾驶台，凭经验和直觉决定是否改航线、换更精确的局部海图
- **L3（最外层）** = 港口气象站和归航船只：每隔几天带回真实海况，船长据此修正最终目的地

## 核心洞见：瓶颈迁移

内层跑得越快，外层（慢反馈）的质量就越决定成败。瓶颈不会消失，只会沿嵌套层次向外迁移——从"代码能否跑通"迁到"spec 是否正确"再迁到"产品方向是否成立"。

如果只升级 L1 而不升级 L2/L3，团队会每天产出大量"看起来完整"的版本，等真实用户反馈回来才发现方向一直没对齐——问题不在 agent，慢反馈没进系统，偏差不会自动消失。

## 核心洞见：上下文优势比品味更工程化

人类留在 loop 里的理由不是玄学的"品味"，而是可证伪的信息不对称：用户是谁、业务约束、场景、竞品、历史踩坑。这给 AI 系统改进指明了路径（让 AI 掌握更多上下文），也给 human-in-the-loop 提供了清晰的退出条件（只要人类比 AI 多知道一点关于用户的关键信息，这一层闭环就必须保留人类）。

## L1 — Agentic Coding Loop（AI 编码循环）

| 维度 | 内容 |
| --- | --- |
| 时间尺度 | 分钟级（每几分钟一轮） |
| 主要驱动者 | AI Agent |
| 核心输入 | Product Spec（产品规格）+ 可选的 Evals（评测集/验收数据集） |
| 核心输出 | 通过验证、无 bug、满足规格的可运行代码 |
| 职责边界 | 只在给定 spec 的边界内优化——把"怎么做"压快，不质疑"该不该做" |
| 典型动作 | 写代码 → 跑测试 → 用浏览器自检 UI/交互 → 发现 bug → 修复 → 再迭代，直到满足规格 |

### L1 工程要点

1. **可演进的 Evals 体系**——不能只有单元测试。必须包含端到端浏览器自动化（Playwright/Puppeteer）、LLM-as-Judge（按明确 rubric 打分用户体验）、关键业务指标的影子测试
2. **停止条件显式化**——目标要可机械验证（"测试全绿"是好目标；"优化代码质量"是坏目标——loop 永远不知何时停）
3. **预算边界**——时间预算、token 预算、尝试次数预算——超预算就停，不为"完成感"继续消耗
4. **执行者/验证者分离**——一个 agent 写，另一个 agent 或脚本查；验证者不能只复述执行者结论
5. **隔离工作现场**——独立 git worktree，避免污染主分支；状态文件记录本轮做了什么、试过什么、从哪继续

### L1 对应机制

| 机制 | 对应实现 |
| --- | --- |
| 确定性控制器 | [controller.md](controller.md) |
| Maker/Checker 分离 | [maker-checker.md](maker-checker.md) |
| 预算/停止条件 | [budget-management.md](budget-management.md) |
| 状态文件 | [state-management.md](state-management.md) |
| Evals/验证 | verification-loop skill + vibe-run-review |
| 工作现场隔离 | using-git-worktrees skill |

### L1 常见失败模式

- **Evals 不完备导致 drift**——agent "聪明地作弊"，修改测试用例让代码通过
- **技术债静默积累**——代码库悄无声息膨胀，无人 review 时快速堆积
- **无进展死循环**——agent 卡在同一错误反复尝试，烧掉几十美元 token
- **理解债（understanding debt）**——人不读 diff，循环越久偏差越大
- **"完整但错误"**——agent 非常认真地把错误规格实现得很完整

## L2 — Developer Feedback Loop（开发者反馈循环）

| 维度 | 内容 |
| --- | --- |
| 时间尺度 | 几十分钟到几小时 |
| 主要驱动者 | 开发者（人类） |
| 核心输入 | 当前产品状态 + 开发者个人愿景/判断 |
| 核心输出 | 更新后的 Spec、方向调整、更清晰的产品愿景 |
| 职责边界 | 把模糊愿景翻译成精确、可执行、可验证的规格；看到实现后迭代规格 |
| 典型动作 | review agent 产出 → 判断功能范围/UI/用户流是否合理 → 改 spec → 交给 L1 继续跑 |

### L2 工程要点

1. **固定节奏**——每 0.5-2 小时一次深度 review，不让内层结果无限堆积
2. **每次 review 只回答一个高价值问题**——方向对不对、体验是否可用、失败是否应变成 eval
3. **Spec 可版本化**——spec 不是 prompt，而是包含目标、非目标、用户场景、不变量、验收证据、权限边界、预算边界、停止条件的可版本化文档
4. **决策记录留痕**——当时为什么这么做、哪些判断后来被证伪、哪些约束还有效
5. **"bridging"能力训练**——把愿景拆成 spec 是可训练、可复用的能力

### L2 设计原则

把"上下文优势"显式注入系统——项目约定、失败案例、实验原则、验证方式必须写下来（AGENTS.md / Skills / Runbook），否则 agent 每轮都重新猜。

### L2 对应机制

| 机制 | 对应实现 |
| --- | --- |
| 可版本化 Spec | spec 文档 + git 版本控制 |
| Review checkpoint | vibe-run-review / review skill |
| 决策记录 | ADR（架构决策记录） + grill-with-docs |
| 非目标显式声明 | spec 模板的 non-goals 字段 |
| Skill 自我进化 | 任务vs系统迭代分离（[self-correction.md](self-correction.md)） |

### L2 常见失败模式

- **人继续当 QA**——agent 自测能力上来后，人只盯 bug 是浪费
- **spec 模糊导致 agent 靠猜补洞**
- **上下文只留在人脑里**——不沉淀到仓库规则/Skill/状态文件
- **方向修正滞后**——等内层堆了大量版本才回头看

## L3 — External Feedback Loop（外部反馈循环）

| 维度 | 内容 |
| --- | --- |
| 时间尺度 | 数小时到数天，甚至数周 |
| 主要驱动者 | 真实用户 / 市场 |
| 核心输入 | 真实使用数据、用户访谈、Alpha/Beta 测试、A/B 实验、竞品动态、支持工单 |
| 核心输出 | 产品愿景迭代、优先级重排、问题是否仍值得做的判断 |
| 职责边界 | 验证"规格背后的产品判断是否站得住"——前两层都发生在系统内部，只有这层碰到真实世界 |
| 典型动作 | 小范围 alpha → 生产 A/B → 用户访谈 → 信号回灌 developer vision → 再驱动 spec 更新 |

### L3 工程要点

1. **低成本采集机制**——结构化用户访谈模板 + 自动化埋点分析，让天级信号更快反哺愿景
2. **反馈必须"进入系统"**——不能停在会议纪要里或被 AI 总结成"用户希望体验更好"——要变成新的产品判断、新的规格约束、新的验收方式、新的非目标
3. **证据强度分级**——来自访谈/工单/埋点/A/B/销售，证据强度不同，影响层级（愿景/规格/验证/文案）也不同
4. **尽早开启**——找几个真实用户试用，比在本地继续修细节更有信息量
5. **AI 辅助但不脑补**——AI 可帮收集、清洗、归纳信号，但信号本身不能靠模型脑补

### L3 设计原则

只要人类比 AI 多知道一点关于用户的关键信息，这一层闭环就必须保留人类。

### L3 对应机制

| 机制 | 对应实现 |
| --- | --- |
| 埋点/分析自动化 | canary-watch skill + 生产审计 |
| A/B 测试基础设施 | 项目级 A/B 配置 |
| 结构化用户访谈模板 | 项目级模板 |
| Feedback 文件 | 记录真实世界回来的信号、证据强度、影响层级 |
| 工单系统接入 | issue-triage 模式 |

### L3 常见失败模式

- **最容易被省掉**——在天级节奏里显得太慢，团队倾向继续打磨实现
- **反馈未结构化**——停留为 anecdote，未变成约束
- **把错误方向做得更精致**——内层循环验证代码是否满足规格，外层验证规格背后的判断是否站得住——前者快后者慢，跳过后者等于在错误方向上加速
- **AI 总结稀释信号**——"用户希望体验更好"这类总结无决策价值

## 三层协同：数据流与控制流

三层是嵌套控制回路，不是流水线。

**正向数据流（愿景→代码）**：

```
Developer Vision → Product Spec + Evals → Coding Agent → 代码+测试 → 当前产品
```

**反向反馈流（证据→愿景）**：

```
当前产品 → Developer Review（L2）→ 更新 Spec/Vision
真实用户/生产数据（L3）→ 修正 Developer Vision → 修正 Spec → 修正 Evals → 下一轮 L1
```

**关键约束**：外部数据路径更长——先影响 developer vision，再影响 detailed spec，最后才驱动 coding agent。这条顺序决定反馈能否进入下一轮。

## 每层交接物契约

每层输出必须能被下一层读懂，这是 ReAct/Plan-Execute 完全不涉及的系统级契约设计：

| 层 | 交接物 |
| --- | --- |
| L1 → L2 | 代码 + 测试 + 日志 + diff + 完成证据 + 残留风险 |
| L2 → L1 | 可版本化 spec + 决策记录 + 非目标 + 验收方式 + 预算边界 |
| L2 → L3 | 当前产品状态 + 已知限制 + 待验证假设 |
| L3 → L2 | 愿景修正 + 优先级调整 + 证据样本 + 影响层级 |
